# sanity-review

PRのレビュー報告書を作成するAgent Skillです。bugや脆弱性の調査だけでなく、exportされた対話コンテキスト・PR概要欄・実装されたコードの整合性を確認し、実装者の正気を疑います。

## 背景

責任共有・知識共有・正確さの向上など、元々コードレビューには様々な役割がありました。コードと実装者とレビュアーという要素間に、AIというアクターが1つ増えたことにより、これはさらに複雑になっています。

実装の正しさだけでなく、実装者が正気を保っているかどうかを、レビュアーは確認する必要が生じています。

- 何をなぜやってどうなったのか、を実装者は説明できているか
- 説明と実装が一致しているか（実装者は、AIが作ったコードを読み解けているのか？）
- bugや脆弱性が含まれていないか
- 実装者の考慮に漏れ・抜けはないか（調べた事、試した事、やらないと決めた事の記述に、十分な厚みがあるか）

## このスキルがやること

上記のチェックリストを調査し、レビュー報告書として出力します。**コードレビューは作業結果のダブルチェックではなく、プロセスのレビューです。** やった事はコードを読めばわかります。検討した上でやらなかった事こそ、設計のレビューに必要な情報です。

対話コンテキスト（`/conversation-context-export` で書き出されたもの）に記録された設計判断・却下理由・意図的な非対応の「プロセス」が正しいかを検証します。

## レビュアー・実装者それぞれのメリット

**レビュアー**: 機械的にチェックできる箇所はAIに任せ、人間がやる必要がある作業に集中できます。例えば、本当にこの変更を受け入れるかどうかの判断、ユーザーやビジネスサイドへの説明、アプリケーション全体としての整合性確認など。

**実装者**: このスキルを自ら使うことで、自分のpull requestを客観的に見直すことができます。

## 前提条件

- GitHub CLI（`gh`）がインストール済みで認証済みであること
- `codex-consultation` スキル: Codexとの相談によるバグ・脆弱性調査と整合性確認に使用（未インストールの場合はAgent単独で続行）
- `conversation-context` スキル: 対話コンテキストの形式の背景知識（未インストールでも動作可能）

## 使い方

```bash
/sanity-review [PR-URL-or-number]
```

PRブランチで実行するか、PR URLまたは番号を指定すると、レビュー報告書を作成します。
