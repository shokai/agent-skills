# チェックリストとレポートの責任境界

観点リスト（checklist）と調査レポート（report）で役割を分ける。
checklistがbugを先取りすると、調査者や手順5の外部Agentがその箇所に引っ張られ、本当に見るべき別のbugを見落とす。
checklistは調査の足場であって、発見リストではない。

このガイドラインは、SKILL.mdの手順2（生成）・手順4（更新）・手順5（外部Agentの反映）から参照される。

## checklist（`CHECKLIST.md` マスター / `.dev/codepatrol/checklist.md` 作業用）

役割: このリポジトリのセキュリティ機構の説明と、中立な観点（何を確認するか）。

- 書く事
  - 仕様・機構の事実。具体名（middleware名・関数名・ファイル・ロール名・環境変数等）を伴う
  - 中立な観点。「〜になっていないか」「〜を確認する」という問いの形
  - 機構が弱く見えても「Xという機構がある / ない」という事実記述（例:「`sanitizeFilter` 未使用」「private-IPブロック無し」はOK）
- 書かない事
  - 特定のbug・脆弱性の断定（「〜で突破できる」「〜が漏れる」）
  - 悪用可否の仮説（中立な問いを超えて「悪用できないか」と一方向に誘導する記述）
  - 深刻度（High / Medium / Low）
  - 既存レポートの発見への言及・引用
  - 「危険」「要注意」「要重点確認」等、特定箇所へ過剰注目を促す表現

## report（Cosenseページ / ローカル `.dev/codepatrol/{領域名}.md`）

役割: 実際の調査結果。発見・非該当・未確認をここに書く。bugはここにだけ書く。

- 書く事
  - 発見したbug・脆弱性、具体的なコード箇所（ファイルパス:行番号）
  - 深刻度と根拠、推奨対応
  - 悪用シナリオ・成立条件
  - 外部Agent（Codex等）の指摘のうち、自分でコードを読んで検証した妥当なもの
- 書かない事
  - リポジトリ全体の機構の網羅的説明（それはchecklistの役目。レポートは調査対象領域の範囲に絞る）

## 判断基準

「これは仕様の説明か、それとも発見か」で分ける。

- **コードを読む前から言える**（仕様・機構・観点）→ checklist
- **コードを読んで初めて言える**（このコードのこの箇所にこの問題がある）→ report

深刻度が付くもの、「悪用できる／突破できる／漏れる」と述べるものは、例外なくreport。

## 混入していたら（浄化）

checklistを生成・更新する時、および既存checklistを使う前に、この境界に照らして全体を監査する。
bug断定・深刻度・悪用仮説・レポート発見への言及を見つけたら、次のいずれかにする:

- 仕様事実に書き直す（例: 「〜で突破できる【High】」→ 「〜という機構である」）
- 中立な観点に書き直す（例: 「危険なので要重点確認」→ 「〜になっていないかを確認する」）
- 観点としても不要なら削除する（発見自体はreportに書く／既にあるなら残す）

観点の追加・更新と、この浄化は、同じ更新作業の中で一緒に行う。

## SSoT

この境界の定義はこのファイルだけに置く。各手順書はここへ参照リンクで繋ぎ、要点の一文だけを添える。
定義を手順書側に複製しない。
