## 1. カッコ書きで補足説明しない

### 悪い例

```markdown
- 説明と実装が一致しているか（実装者は、AIが作ったコードを読み解けているのか？）
- 実装者の考慮に漏れ・抜けはないか（調べた事、試した事、やらないと決めた事の記述に、十分な厚みがあるか）
- `codex-consultation` スキル: Codexとの相談によるバグ・脆弱性調査と整合性確認に使用（未インストールの場合はAgent単独で続行）
```

### 良い例

```markdown
- 説明と実装が一致しているか
  - 実装者はAIが作ったコードを読み解けているのか
- 実装者の考慮に漏れ・抜けはないか
  - 調べた事、試した事、やらないと決めた事の記述に十分な厚みがあるか
- `codex-consultation` スキル: Codexとの相談によるバグ・脆弱性調査と整合性確認に使用
  - 未インストールの場合はAgent単独で続行する
```

### 良い例: 文を分けて続ける

```markdown
悪い: 対話コンテキスト（`/conversation-context-export` で書き出されたもの）に記録された設計判断・却下理由・意図的な非対応を検証する。
良い: `/conversation-context-export` で書き出された対話コンテキストには、設計判断・却下理由・意図的な非対応が記録されています。このスキルは、それらの「プロセス」が正しいかを検証します。
```

### 良い例: 文の構成を推敲する

```markdown
悪い: レビュアーは機械的にチェックできる箇所はAIに任せ、人間がやる必要がある作業（判断、説明、整合性確認など）に集中できます。
良い: レビュアーは機械的にチェックできる箇所はAIに任せ、人間がやる必要がある作業に集中できます。

- 本当にこの変更を受け入れるかどうかの判断
- ユーザーやビジネスサイドへの説明
- アプリケーション全体としての整合性確認
```

### 説明

丸括弧 `（）` で補足説明を挿入すると、文の構造が曖昧になり読みにくくなる。補足の内容が本文に埋もれ、読み手は括弧の中と外を行き来しながら意味を解釈する必要がある。

代わりに以下の手法を使う:

- **子リストで展開する**: 補足内容が親の項目の詳細や判断基準である場合。入れ子のリスト記法を使い、階層構造として明示する
- **文を分けて続ける**: 補足内容が背景説明や理由である場合。句点で区切り、独立した文として書く
- **文の構成自体を推敲する**: 括弧の中身を先に述べてから本題に入るなど、文の順序を変えることで括弧が不要になる場合がある
