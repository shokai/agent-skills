# 定石外発想によるリファクタリング手順書

実装済みの解（第一案）が依存している前提条件を1つずつ外し、よりシンプルな別解がないかを検討する。

## このスキルが解決する問題

AIに問題解決を頼むと、どうしても「現状の発展型」しか出てこない。

- Skillを改善させると、記述を足してカバーする方向に行く
- プログラムを改善させると、機能を足す方向に行く
- ソフトウェア開発の定石に素直に従いすぎて、過剰な実装になる

人間は並行して別のことをしている。既知の定石を連想しつつ、同時に「枷＝前提条件」を1つずつ外し、「この前提が無ければもっと簡単に解けるのではないか」という当たりを付けている。このスキルは、その人間がやっている手順をAgentに実行させるためのものである。

`批判的思考の連鎖` で「やり方や前提まで疑え」と指示しても足りない。bug修正ならともかく、問題解決において前提を疑う _方法_ をAgentは持っていない。このスキルはその方法を手順として与える。

## 使いどころと位置づけ

このスキルは **二段ループの後段ツール** である。

1. まずAgentが指示通りに素直に実装する。これが第一案であり、たいてい定石的な解になる
2. その第一案を見てから、このスキルで枷を外して再考する

定石外発想を常時オンにするとノイジーになる。Agentはまず素直に作業すべきであり、このスキルは求められた時だけ発動する。

利用シーンは実装後のリファクタリングである。新機能を足すためではなく、すでにある解をよりシンプルにできないか探すために使う。

## 手順1: 分析対象の特定

「いま再考すべき解（第一案）」を特定する。

1. **引数がある場合**: 指定されたファイル・モジュール・機能を対象とする
2. **会話コンテキストがある場合**: 直前に到達した方針・設計判断・実装アプローチを対象とする。これが主たる特定方法である
3. **コールドスタートの場合**: 会話コンテキストが薄く引数も無いときは、直近の変更を自動検出して対象とする。何を対象と推定したかを必ず一言で明示してから先に進む

対象を特定したら、その解を実装しているコードを以下の順で確認し、前提を裏付ける:

1. 引数で対象ファイル・モジュールが指定されている場合は、まずそれを直接読む
2. 未commitの変更を `git diff` と `git diff --staged` で確認する
3. 未commitの変更が無ければ、base branchとの差分や直近のcommitを確認する。base branchは `git symbolic-ref refs/remotes/origin/HEAD` 等で判定し、例えば `git diff origin/main...HEAD` のように差分を取る。判定できなければ `git log -p -3` で直近のcommitを見る
4. 会話で名前が挙がったファイルがあれば直接読む

commit済みかどうかには縛られない。再考対象の解が表れているコードを掴むことが目的である。

## 手順2: 前提条件の列挙

採用された解が暗黙に置いている前提条件を、コードと会話コンテキストから列挙する。

前提条件の例:

- 「この処理はリアルタイムでなければならない」
- 「既存のAPIとの互換を保たねばならない」
- 「このデータ構造を使う」
- 「この状態を保持し続ける必要がある」
- 「この機能はユーザーから見える形でなければならない」

問題設定そのものへの一段メタな前提も列挙対象に含める。「そもそもこれはどういうユーザー・状況なのか」「もっと手前の段階で対処すべきではないか」というところまで戻って前提を疑う。

**重要**: コードと会話コンテキストに根拠のある前提だけを扱う。空想上の要件・前提を持ち込まない。レビュー対象の解が実際に依存している前提だけが対象である。

## 手順3: 前提を1つずつ外して再考

列挙した前提を1つずつ取り上げ、「この前提が無ければどういう解になるか」を考える。

- 前提を外してもシンプルにならないものは、その旨と理由を軽く記録してスキップする
- 前提を外すとシンプルになるものは、その別解を具体化する

一度に複数の前提を外さない。1つずつ外すことで、どの前提が効いているのかを切り分ける。

## 手順4: コスト比較

シンプルになった別解について、定石案である第一案と比較する。

以下の5観点で比較する:

1. **実装量**: 第一案と比べてどれだけコードが減るか
2. **既存仕様との互換性**: 既存の動作・APIとの互換が壊れないか
3. **ユーザー・開発者への説明コスト**: 概念や挙動を説明するのが難しくならないか
4. **運用・移行コスト**: 既存データや運用フローの移行が必要になるか
5. **将来の保守性**: 長期的に保守しやすくなるか、禍根を残さないか

加えて、前提解除を **成立させるために必要な別の仕組み** を明示的に計上する。前提を外せた裏で、別の追加実装・ドキュメント・ユーザーへの教育などが必要になっていないか。簡潔さを得た代わりにどこにコストが移動したかを正直に見積もる。

## 手順5: 分類

各別解を3つに分類する。

- **今すぐ検討すべき**: シンプルさの利得が大きく、コストが見合う。第一案を置き換える価値がある
- **将来の論点**: いまは第一案で妥当だが、状況が変われば再考に値する
- **現状の採用案で妥当**: 前提を外してもメリットが薄い、またはコストが上回る

## 手順6: 外部Agent相談（任意）

このスキルは既定では単独で実行する。外部Agentへの相談は任意であり、ユーザーが外部Agentとの相談を明示的に求めた場合にのみ行う。「codexと相談して」「subagentと相談して」「軽く相談して」のように、外部Agentへの相談意思が示されたときである。分析の深さの指定と外部相談の指定は別物として扱い、相談意思が示されないときは単独で実行する。

相談する場合は、以下のフォールバック順序に従う:

1. Skill toolで `codex-consultation` を呼び出す。失敗した場合は2へ進む
2. Skill toolで `subagent-consultation` を呼び出す。失敗した場合は3へ進む
3. main agentが単独で作業を実行する

**重要**: 外部Agentは放っておくと既存案のレビュー、すなわち現状の発展型の指摘に流れる。「レビューして」ではなく、手順2〜5そのもの、つまり前提を自分で列挙して1つずつ外しよりシンプルな解を探す作業を実行させること。別の視点で前提を洗い出させることで、Agentが当然視して見落とした前提を拾える。

相談時のArgs例:

```
Args: 軽く相談して。次の採用案が依存している前提を自分で列挙し、各前提を1つずつ外したらよりシンプルな解になるか検討してほしい。既存案のレビューではなく前提の組み替えを求めている。コードと文脈に根拠のある前提だけを扱い、空想上の要件は持ち込まないこと。{採用案と直近変更の要約}
```

外部Agentの別解を受け取ったら、自分の再評価と照合し、見落とした前提やより良い解がなかったか確認する。外部Agentの指摘を鵜呑みにせず、自分でコードを読んで検証する。

## 手順7: 報告

結果を会話に出力する。ファイルには保存しない。

このスキルは **提案・分類で停止する**。コードの自動適用はしない。どの別解を採用するかは人間が判断する。

報告フォーマット:

```markdown
## 定石外発想によるリファクタリング検討

### 対象

{再考対象の解の要約。コールドスタート時は何を対象と推定したかも明記}

### 列挙した前提条件

1. {前提1}
2. {前提2}
   ...

### 各前提の再考結果

#### 前提1: {前提の内容}

- **外したときの別解**: {具体的な別解。シンプルにならない場合はその旨と理由}
- **5観点比較**: 実装量 / 互換性 / 説明コスト / 運用移行コスト / 保守性 のうち効くものを記載
- **前提解除に必要な別の仕組み**: {追加実装・教育など。無ければ「なし」}
- **分類**: 今すぐ検討すべき / 将来の論点 / 現状の採用案で妥当

#### 前提2: {前提の内容}

...

### 推奨アクション

{「今すぐ検討すべき」に分類した別解があれば、何を検討すべきかを示す。無ければ「現状の採用案で妥当」と結論する}
```

## 関連スキル

- **codex-consultation**: Codex CLIと相談するスキル。手順6で外部Agentの第一候補として任意で使用する
- **subagent-consultation**: Agentツール（subagent）と相談するスキル。codex-consultationが利用できない場合のフォールバック先
- **sanity-review**: PRレビュー報告書を作成するスキル。やった事ではなく検討プロセスをレビューするという思想を共有する。sanity-reviewが却下した代替案や失敗した試行が依存する前提を疑うのに対し、このスキルは採用された解そのものが依存する前提を外してよりシンプルな別解を探す。実装後のリファクタリングに特化した位置づけである
